# Clody

Реалтайм-мессенджер с голосовыми звонками, демонстрацией экрана и каналами-«пикниками».
Бэкенд — Flask + Flask-SocketIO, фронтенд — React (Vite) SPA. Весь интерфейс на русском.

---

## Возможности

- **Лавочки** — переписки: личные (1-на-1) и групповые, с вложениями (картинки, видео, файлы),
  ответами на сообщения, `@упоминаниями` и форматированием (`**жирный**`, `*курсив*`).
- **Звонки и демонстрация экрана** — голосовой звонок в лавочке, трансляция экрана. Медиа идёт
  через отдельный voice-сервер (бинарный релей, без WebRTC).
- **Пикники** — каналы в духе Telegram: посты пишут только админы, остальные читают. Пикник
  публичный — его можно найти поиском и читать, не вступая.
  - **Комментарии** под постами: лайки, ответы, редактирование, удаление.
  - **Анонимность**: если выключить «Разрешить отправлять мне запросы в друзья», в комментариях
    вы анонимны — вас не раскроют даже админы, только ваши друзья.
  - Кто владелец и админы пикника скрыто от обычных участников.
- **Друзья и враги**, заявки в друзья, профили.
- **Ленивая подгрузка** сообщений, постов и комментариев (грузятся частями по мере прокрутки).
- **Модерация**: жалобы на сообщения, предупреждения, админ-панель.

**Бэкенд**

## Установка

```bash
# 1. Бэкенд: виртуальное окружение и зависимости
python -m venv .venv
.venv\Scripts\pip install -r requirements.txt

# 2. Фронтенд: зависимости
cd frontend
npm install
cd ..
```

Создайте файл `.env` в корне проекта:

```env
SECRET_KEY=любая-длинная-случайная-строка
RECAPTCHA_TOKEN=секретный-ключ-google-recaptcha
```

- `SECRET_KEY` — подписывает сессии и HMAC-токены voice-сервера. Должен быть одинаковым для
  основного приложения и voice-сервера (они его читают из общего `.env`).
- `RECAPTCHA_TOKEN` — секрет reCAPTCHA, проверяется при регистрации.

## Запуск

### 1. Собрать фронтенд

Flask раздаёт **собранный** фронтенд из папки `static/` по адресу `/app/`. После любого
изменения фронта пересоберите:

```bash
cd frontend
npm run build      # результат кладётся в ../static/
cd ..
```

### 2. Запустить бэкенд

```bash
.venv\Scripts\python app.py
```

- Слушает `http://127.0.0.1:20001`, приложение открывается по **`http://127.0.0.1:20001/app/`**
  (корень `/` — не приложение).
- `app.py` автоматически поднимает **voice-сервер** (`voice_server.py`) на порту `20002`
  отдельным процессом. То есть `python app.py` = два процесса.
- В debug-режиме бэкенд перезагружается на изменения `.py`.

### Dev-режим фронтенда (опционально)

Для горячей перезагрузки при разработке фронта:

```bash
cd frontend
npm run dev        # порт 5173, проксирует /api → :20001
```

> Realtime (сокеты, звонки, presence) требует запущенного основного бэкенда. Для проверки
> звонков и реалтайма надёжнее тестировать собранную версию на `:20001/app/`.

## Архитектура

### Сессии и авторизация

- Сессии — cookie Flask, ключ `session["user"] = {id, login_at}`.
- `check_session()` в `ext.py` — единая проверка авторизации во всех роутах и обработчиках сокетов.
  Поддерживает «выход со всех устройств» через `data["logout_devices_at"]`.
- **Один браузер = одна сессия.** Двух разных пользователей в двух вкладках одного браузера держать
  нельзя (общий cookie + presence-guard). Для мультиюзер-сценариев используйте разные браузеры/профили
  или гоняйте API напрямую через `requests.Session()`.

### Модель данных (`db.py`)

Таблицы: `User`, `Branch` (лавочка) + `BMessage`, `Picnic` (пикник) + `PMessage`, `Comment`, `Report`.

Особенность: часть изменяемого состояния лежит в сериализованных колонках.
- **`User.data`** — JSON-**строка** с `{online, branches[], friends[], enemies[], picnics[], settings{}, ...}`.
  Читать через `load_data(user)`, писать `json.dumps` обратно в `user.data`.
- **`PickleType`-колонки** (`Branch.members/messages`, `BMessage.read/content/cdn`, `Picnic.members/admins/bans`,
  `Comment.likes` …) **не отслеживают мутации на месте**: `list.append(...)` молча не сохранится.
  Всегда переприсваивайте целиком (`msg.read = (msg.read or []) + [uid]`) и делайте `db.session.commit()`.

### Реалтайм (два namespace Socket.IO)

- **Основной namespace** — presence и адресные события пользователю (`new_bmessage`, `new_pmessage`,
  `new_comment`, `friend_request`, `new_call`, …).
- **`/bcalls`** — только сигналинг звонков (роутинг, mute, кто демонстрирует экран, звуки).
  Состояние звонков живёт в памяти процесса и сбрасывается при рестарте.

### Медиа на отдельном voice-сервере

Голос и экран передаются бинарными кадрами по отдельному Socket.IO-соединению к
`voice_server.py` (порт `20002`) — это простой релей (без WebRTC/SFU/aiortc). Авторизация между
процессами — короткоживущий HMAC-токен (`voice_token.py`, подписан общим `SECRET_KEY`).

### Пикники, комментарии и приватность

- Пикник **публичный**: любой авторизованный может читать посты и комментарии; писать посты —
  только админы, комментарии — только участники.
- Сводка пикника (`picnic_summary`) не раскрывает списки админов/владельца/участников — отдаёт
  только флаги про самого зрителя (`is_member`, `is_admin`, `is_owner`) и `members_count`. Управление
  ролями — через админский роут `/api/picnic/manage`.
- Автор поста виден только админам; для остальных пикник «говорит одним голосом».
- Анонимность комментариев: `visible_author()` в `ext.py` отдаёт автора только если зритель — сам
  автор, автор не анонимен, или зритель у него в друзьях.

### Ленивая подгрузка

- `/api/bm/list`, `/api/pm/list`, `/api/comment/list` принимают `before_id` + `limit` и возвращают
  `{messages|comments, has_more}`. Сообщения/посты грузятся страницами при прокрутке вверх, комментарии —
  кнопкой «Показать ещё».
- Счётчики непрочитанного и время активности берутся одним запросом: `/api/branches/overview` и
  `/api/picnics/overview` (вместо N запросов на каждую лавочку/пикник).

## Структура

```
app.py               точка входа: Flask, регистрация блюпринтов, запуск voice-сервера
voice_server.py      отдельный процесс-релей медиа (порт 20002)
voice_token.py       HMAC-токены для авторизации на voice-сервере
db.py                модели SQLAlchemy
ext.py               socketio, check_session, load_data, анонимность, системные сообщения
ratelimit.py         ограничение частоты запросов
migrate_db.py        миграция схемы (добавляет недостающие колонки существующим таблицам)
blueprints/          роуты: login, friends, branches, bmessages, picnics, pmessages,
                     comments, cdn, settings, bcall, adminpanel, static
frontend/            React SPA (Vite), собирается в ../static/
static/              собранный фронтенд, который раздаёт Flask (/app/)
cdn/                 загруженные файлы (аватары, вложения лавочек и пикников)
```

## Миграции схемы

Отдельной системы миграций (Alembic) нет: `db.create_all()` только **создаёт отсутствующие**
таблицы, но не изменяет существующие. Если модель обрастает новыми колонками, старую таблицу
на сервере нужно догнать вручную:

```bash
.venv\Scripts\python migrate_db.py                 # instance/database.db рядом со скриптом
.venv\Scripts\python migrate_db.py /путь/database.db
```

Скрипт делает резервную копию БД и добавляет недостающие колонки (`picnic`, `comment`).
Безопасно запускать повторно. **Запускать при остановленном приложении.**
